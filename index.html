01

Executive Summary

Over 18 months, I served as the sole product designer on Octopus ERP — a modular platform built to replace the fragmented toolsets that Iranian mid-market businesses operate with daily. My ownership covered the full design lifecycle: zero-to-one design system, complete per-module design packages, UX writing, and two versions of the marketing website.

This case study focuses on the core challenge: making 28+ modules feel like one coherent system to users who work across all of them — in Persian and English — without the friction of switching between disconnected tools.

The honest scope: I designed or redesigned 28 modules with varying depth. Roughly half reached full high-fidelity and developer handoff under my ownership. The rest progressed to validated wireframes or partial high-fidelity, with implementation continuing beyond my primary involvement. The IA section below makes this distinction explicit.

02

Business Context

Mid-market businesses in Iran operate with fragmented toolsets — separate accounting software, standalone HR systems, disconnected inventory trackers. The real cost is invisible: manual data reconciliation between tools, delayed approvals because context lives in email, and stockouts that happen because Inventory and Procurement don't talk.

Octopus was built to replace this. Every module connects to a shared data layer. A purchase approval in one module automatically updates Finance and Inventory. The platform's value proposition is the connection between modules — which meant the design system had to make those connections effortless, not just possible.

The design challenge wasn't building 28 modules. It was making them feel like one.

Problem Statement

Enterprise users managing cross-department workflows lose meaningful time when navigating between disconnected, inconsistent modules. Every tool that feels different is a moment of hesitation — and in an ERP used 8 hours a day, hesitation compounds into real cost.

Three root problems — and what we actually knew

  • 01

    No shared design language — each module felt like a different product

    Early module prototypes were built independently. By the time I joined, Finance and Procurement already had divergent nav patterns, different button hierarchies, and inconsistent spacing. Users in PM-led feedback sessions consistently described the experience as "switching between apps." This was observed, not assumed.

  • 02

    Data connections were technically real but experientially invisible

    A Finance Controller had no way to know that the record they were approving was created by a Procurement request 3 days ago. The data existed. The story didn't. We identified this gap in a PM-led customer discovery call with two Finance users from a pilot company — both asked "where did this come from?" within the first 10 minutes of using the system.

  • 03

    RTL support was bolted on, not designed in

    Persian is the primary language for our users, but the early codebase treated RTL as a translation layer — flipping text direction but not layout logic. Icons didn't mirror. Form field order didn't reverse. Sidebar stayed left. This was a structural problem, not a CSS fix.

The Core Workflow: Purchase to Payment

Rather than mapping all possible journeys, this case study goes deep on one: a purchase request traveling from creation to payment, crossing Purchase, Accounting, and Inventory. This was the highest-friction workflow identified in discovery — and the clearest test of whether the platform's inter-module data promise actually felt real to users.

1

Request Creation — Purchase Module

Operations Director creates a purchase request. Budget validation queries Accounting in real time, inline — result appears next to the amount field, not in a separate screen.

Previous friction: 3 screens to validate budget
2

In-app Approval — Accounting

Finance Controller receives a contextual notification inside Accounting with the PO details, origin module badge, and requestor — fully pre-populated. One action to approve.

Previous friction: email notification → separate login → manual re-entry
3

Inventory Auto-update

Approval triggers an expected stock arrival entry in Inventory. Warehouse Supervisor sees it as a pending receipt — no manual entry, no phone call from procurement.

Previous friction: manual WhatsApp message from procurement
4

Goods Receipt — Inventory

Supervisor confirms receipt. System triggers three-way match (PO ↔ receipt ↔ invoice) in Accounting, auto-flagging discrepancies rather than passing them silently.

5

Payment — Accounting / PettyCash

Finance Controller completes payment. The full workflow timeline — from original request to payment — is traceable from any module that touched it.

Key Decisions — and What We Rejected

Every decision below went through at least two alternatives. The rejected paths are part of the work.

How to handle RTL layout across all modules
Option A — Rejected
A separate RTL stylesheet per module, maintained alongside the LTR version. This was the existing approach in the early codebase.
Option B — Rejected
A global CSS class (.rtl) that overrides left/right properties. Fast to implement, but creates a maintenance debt every time a new component is added.
Option C — Chosen
CSS logical properties (inline-start, inline-end, block-start) used as the default for all layout tokens. RTL/LTR becomes a single HTML dir attribute — no overrides, no parallel sheets. Aligned with Zahra (frontend) in a dedicated session before the component library was built.

Why the others were rejected: Option A doubles maintenance burden at 28+ modules. Option B creates invisible technical debt — any dev adding a "left: 12px" breaks RTL silently. Option C costs more upfront but removes the failure mode entirely.

How to show cross-module data provenance
Option A — Rejected
Persistent source label on every record ("Created via Procurement · 3 days ago"). Tested in a low-fidelity walkthrough with 3 internal users — all three reported the labels as "noise" when looking at a table of 20 records. Abandoned after one session.
Option B — Rejected
A dedicated "history" tab inside each record. This placed the provenance story behind a click — which meant users would only find it if they already knew to look.
Option C — Chosen
Hover-triggered source badge showing origin module icon + timestamp. Visible on demand, invisible by default. Combined with a persistent audit trail accessible from the record's action menu.

The walkthrough test with 3 users wasn't statistically significant — but it gave directional signal fast. The persistent label caused every participant to comment on visual clutter before completing the primary task. That was enough to rule it out.

Gantt / Timeline direction in RTL mode
Option A — Rejected
Mirror the entire timeline axis in RTL — earliest date on the right, latest on the left. Consistent with RTL layout logic.
Option B — Chosen
Keep timeline left-to-right even in RTL mode. Time flows left to right universally in every calendar system our users reference (Jalali included). Mirroring the time axis created genuine confusion in a walkthrough — users looked for "today" on the right and couldn't orient themselves.

This was a deliberate exception to the mirroring rule — documented explicitly in the design system as "temporal axes are exempt from RTL mirroring." Making exceptions is fine; undocumented exceptions are not.

Navigation structure — how users move between modules
Option A — Rejected
Module-level top navigation (each module has its own full nav bar). Tested in wireframes — created the perception that modules were separate products. The very problem we were solving.
Option B — Rejected
Full mega-menu with all 28+ modules listed at once. PM pushed for this initially to showcase breadth. I argued it front-loads cognitive load on users who only use 4–6 modules daily.
Option C — Chosen
Persistent global shell (always visible) + module-specific secondary sidebar. Users experience a stable "home" layer and a contextual "where I am" layer simultaneously. The disagreement with PM on the mega-menu was resolved by proposing a module favorites system — users pin the modules they use, so breadth is accessible but not forced.

The PM's concern was valid — enterprise buyers evaluate software by feature count. The favorites system satisfied both the user experience goal and the sales demo goal. Neither of us got everything we wanted, which is probably the right outcome.

Known gap — danger color: textColor.danger (#EF4444) at 3.9:1 passes AA for large text only. We use it exclusively for icon-accompanied error labels (never standalone small text) as a documented exception. Replacing it with a darker red would conflict with the strokeColor.red token. This trade-off is logged and will be revisited in the next token audit.

Keyboard & screen reader

  • —

    All interactive components include keyboard focus states

    Focus rings use a 2px offset stroke in strokeColor.default. Documented as a required variant in every interactive component in the design system.

  • —

    Semantic HTML specified in handoff docs

    Every component annotation includes the correct HTML element (button vs. div, role="alert" for notifications, aria-label for icon-only buttons). Enforced in bi-weekly handoff sessions with Zahra and Amir Hosein.

  • —

    RTL screen reader order

    In RTL mode, DOM order matches visual reading order. This is a constraint of the logical properties approach — by using inline-start for layout, the visual and DOM order stay aligned without manual aria-flowto overrides.

Impact & Outcomes

I don't have post-launch conversion or revenue numbers to share — Octopus was in active rollout at the end of my involvement, and those metrics belong to a longer timeline than this engagement covered. What I can speak to honestly:

28+
Modules with design coverage across 7 verticals
2
Languages with full bidirectional support at launch
0
Parallel RTL stylesheets — logical properties only

  • Cross-module friction was the top complaint in discovery — and was directly addressed

    The specific frustrations captured in PM-led discovery calls ("five tabs open to understand what's happening," "where did this PO come from") were the design targets. The persistent global shell, contextual notifications, and source badges were direct responses to those exact quotes.

  • Design–dev rework dropped noticeably once the token system was in place

    Before bi-weekly handoff sessions and the shared token library, frontend would return design questions mid-sprint. After the system was established, the majority of sprints completed without design revision requests. Amin (PM) noted this explicitly in a sprint retro.

  • New module onboarding time decreased substantially

    The first 5 modules required significant design investment per module. By module 10, the team was reusing 70–80% of existing components. By module 15+, new modules were primarily IA and content work — the visual design was already solved. This isn't a guess; it's visible in the Figma commit history.

  • Website v2 pricing configurator gave the sales team a self-serve demo tool

    The modular pricing configurator meant prospects could assemble their own module bundle and see pricing update in real time — reducing the number of discovery calls needed before a quote. This was noted as a workflow improvement by the sales team in a post-launch review.

Lessons Learned

A real conflict — and what it taught me

Conflict — PM vs. Designer

The mega-menu argument

Around month 4, Amin (PM) pushed to include all 28+ modules in a full mega-menu in the global shell. His reasoning was clear: enterprise buyers evaluate software by feature breadth, and the sales demo needed to show everything at once.

I pushed back. A mega-menu front-loads cognitive load on users who use 4–6 modules daily. It's a sales artifact, not a UX artifact. But I also recognized that Amin's concern was legitimate — you can't ignore the sales context for a B2B product.

Resolution: I proposed a module favorites system — users pin their most-used modules to the primary nav, and the full library is accessible one level deeper. The mega-menu exists as a discovery surface during onboarding and demo, not as the daily navigation layer. Neither of us got our first choice. Both goals were met.
Constraint — No dedicated researcher

What I did instead — and where it was insufficient

Without a researcher, I ran informal walkthroughs, observed PM discovery calls, and synthesized support ticket themes. This gave me directional signal quickly, but it's not the same as structured research.

The persistent label test (3 users, one session) was enough to rule out an option. It wasn't enough to be confident about what the right replacement was. I made a judgment call — hover badge — and it held up in subsequent walkthroughs. But if I had research support, I would have tested 3 alternatives instead of 2, with more users, over more sessions.

What I'd do differently: involve QA (Negar) earlier as an informal research partner. She had structured access to users during testing cycles that I could have observed more systematically. I didn't fully use that resource until month 9.

What I'd protect

  • ✓

    CSS logical properties from day one

    The investment in aligning Zahra on logical properties before writing a single component saved us from what would have been a painful RTL retrofit across 28 modules at month 8. The upfront alignment conversation was 3 hours. The retrofit would have been weeks.

  • ✓

    Documenting deliberate exceptions to the design rules

    The timeline axis RTL exception is easy to implement wrong if a future developer doesn't know it's intentional. Writing it into the design system as a named exception — not just implementing it — meant it survived handoffs.

  • ✓

    Making the accessibility gap visible, not hiding it

    The danger color contrast issue is a real problem. Logging it as a known, documented exception — with a rationale and a future review date — is more honest than either ignoring it or over-engineering a fix that breaks other tokens. Known problems are manageable. Invisible problems aren't.

What I'd do differently

  • △

    Design cross-module patterns before module IA — not after

    The notification system and audit trail were designed after 6 individual module IAs were already set. Retrofitting them required minor IA adjustments in 4 modules. If I'd designed the "connective tissue" first and treated individual modules as surfaces on top of it, that rework disappears.

  • △

    Establish a baseline early — even with a low-fidelity prototype

    I didn't run a structured baseline task test until month 6. By then, I'd made 30+ decisions without a comparable "before" state. I have directional signal but no before/after data that would be defensible to a research-focused audience. Sprint 1, even with paper prototypes, would have been worth it.

An ERP is not a product you choose to use. It is the product you live inside. The job of design here is not to delight — it is to disappear. The best outcome is a user who finishes a workflow and never thinks about the interface at all.
